## Key Architectural Decisions

### âœ… Why Database Triggers?
1. **Reliability:** Server-side execution, no client timing issues
2. **Security:** Elevated privileges, bypasses RLS
3. **Atomicity:** Profile creation happens in same transaction
4. **Consistency:** Every user automatically gets a profile
5. **Simplicity:** Client code is simpler and cleaner

### âœ… Why SECURITY DEFINER?
- Allows trigger to bypass RLS policies
- Executes with privileges of function owner
- Safe because function is controlled and audited
- Standard pattern for user provisioning

### âœ… Why 1-Second Delay?
- Safety margin for trigger to complete
- Could be reduced to 100-200ms for production
- Ensures profile exists before loading
- Better than retry loops

### âœ… Why No INSERT Policy on profiles?
- Trigger handles all profile creation
- Users should never manually insert profiles
- Prevents duplicate or malformed profiles
- Cleaner security model
